## 0. 整體評語

這份 v0.2 需求文件以「單人開發者用個人交易系統」而言，**完整度與成熟度都相當高**：
- 有清晰的目標 / 非目標、利害關係人、UC、FR、NFR、資料模型、流程圖、風險、里程碑、開放議題、技術棧。
- 安全設計（兩段確認、四大護欄）有亮點，反映出對「誤觸 LIVE 損失真錢」的高度警惕。
- 把 LLM 新聞模組獨立為 M5.5、明確規定「絕不自動下單」── 對 AI 幻覺與 pump-and-dump 風險的防範意識很到位。

**但作為一份「可進入設計階段」的文件**，仍有 14 項模糊地帶 / 缺漏 / 矛盾 / 時程過於樂觀的問題，下面逐條展開。

## 1. 發現的問題（issues）

### I-1 模式切換時，模擬部位 / 訂單該如何處理？── **嚴重缺漏**
**問題：** 假設使用者已在模擬模式持倉 QQQ 5 股、SPY 3 股，按下「切換到實盤」並通過二次確認。切換完成後：
- 模擬部位是否消失（被「LIVE 帳本」取代）？
- 還是保留模擬部位，只是不再出現在 LIVE 主控台？
- 或兩套帳本平行存在，主控台依當前模式只顯示對應的部位？

MD 第 7.1 表的 `accounts` 設計成「模擬 / 實盤分開兩列」── **暗示是兩套並存**，但 UC-04、第 10.1 流程、FR-MM-* 都沒明說切換時 UI 該如何顯示、24h 自動回模擬時持倉如何「回到模擬端」。

更尖銳的問題：
- 24h 後自動 `SimulationModeRestored` 時，使用者在 LIVE 期間建立的真實持倉怎麼辦？仍然顯示？標示為「鎖定，非當前模式」？
- 美股 LIVE 時寄出了訊號 Email，使用者尚未手動下單，這時系統自動回 SIM 模式 ── 該訊號狀態如何處理？

**建議行動：** 補 FR-MM-08 / FR-MM-09，明定：
- 持倉、訂單、訊號表的「mode 隔離」原則：每張表的 `mode` 欄位即為過濾器，UI 依當前 ModeState 過濾顯示。
- 自動回 SIM 時，現有 LIVE 持倉「鎖定不顯示」但資料保留；下次手動切回 LIVE 才恢復可見。

### I-2 「總持倉 ≤ 80%」如何在 SIM / LIVE 之間計算？── **規則模糊**
FR-RM-02 規定「總持倉部位上限 80%，保留 20% 現金」。
- 80% 是針對「當前模式對應的帳戶總值」計算，還是「總資產」？
- 模擬帳戶起始 100,000 TWD / 3,000 USD ── 換匯率怎麼處理？
- 多幣別計算「總持倉」時，TWD 部位與 USD 部位如何加總到統一基準？

**建議行動：** 補 FR-RM-06 明定多幣別折算規則（例如「每日以收盤後的台銀牌告匯率換算為 TWD 計算」），並說明 SIM / LIVE 帳本各自獨立計算 80%。

### I-3 美股訊號的「執行截止時間」未定義 ── **規則模糊**
HTML Email mockup 顯示「執行截止 2026-05-24 美股開盤後 30 分鐘內」。
- 若使用者未在截止時間內手動下單，系統如何處理？
  - (a) 訊號自動失效 → 寫入 signals 表 status=expired？
  - (b) 系統重新運算？
  - (c) 假設使用者已下單，於下次自動同步時對帳？
- 沒回信回報是否要寄催信？

MD FR-EX-02 / 4.6 通知系統都沒提這個生命週期管理。

**建議行動：** 補 FR-EX-06：美股訊號需有完整生命週期狀態（pending → expired / filled / cancelled），並設計同步機制（例如下次 yfinance 拉到價量變動推測使用者已成交、寫回 `filled_at`）。

### I-4 「策略產生的訊號違反風控」的處理粒度 ── **規則模糊**
FR-RM-04 規定違反風控的訊號需被攔截並寫入日誌。但：
- 若一個策略執行產生 5 個訊號，其中 3 個違反風控、2 個通過 ── 系統是「全部攔截」還是「分別處理」？
- 違反風控的原因若是「總持倉已 78%、本次再加碼會超過 80%」，是否要部分縮量（買 0.5 倍）？
- 違反「單日虧損 3%」熔斷時，當天剩餘所有訊號是否全部攔截？包括減倉 / 平倉訊號嗎？

**建議行動：** 補 FR-RM-07 明定：
- 訊號逐筆審查、各自進入後續流程。
- 違反總持倉 80% 時是否允許部分減量（建議：MVP 不減量，整筆拒絕）。
- 單日虧損熔斷時，**只攔截「新進場 / 加碼」訊號，不攔截減碼 / 平倉 / 停損訊號**（MD 11.1 風險表已暗示，但 FR 條文未明言）。

### I-5 LLM 「絕不自動下單」與 FR-CH-27 「形態作為條件過濾器」會不會打架？── **設計矛盾風險**
FR-NS-21 規定新聞模組「絕不」自動產生訊號，必須兩段核可。
FR-CH-27（P2 進階）允許 K 線形態「加入策略訊號的條件過濾器」。

**設想場景：** 使用者啟用 FR-CH-27，把「黃金交叉」設為 Dual Momentum 訊號的必要條件。某日 Dual Momentum 自動產生 IWM 買進訊號，恰好 IWM 那天黃金交叉，於是訊號通過過濾器 → 進入風控 → 自動下單。

這沒有違反 FR-NS-21（沒走新聞路徑），但**「自動形態 + 自動策略 = 自動下單」這條路徑可能讓使用者誤以為形態本身觸發了下單**。需要在 UI 上明確標示「本次訊號通過了哪些過濾條件」。

**建議行動：** 補 FR-CH-28：當形態過濾器啟用且生效時，訊號 Email 與訊號日誌應記錄「通過的過濾器名稱」，方便事後追溯。

### I-6 24 小時自動回 SIM 是「距上次切換」還是「距上次切換到 LIVE」？── **語意歧義**
FR-MM-05 寫「距上次切換 > 24 小時則自動回到模擬模式」。
- 字面意思是：任何切換動作（含 SIM → SIM 重啟、LIVE → SIM 手動切回）都會重置計時器？
- 還是專指「進入 LIVE 後超過 24 小時自動回 SIM」？

從 11.1 風險表「24h 自動重置」推測應為後者，但 FR 條文表述不嚴謹。

**建議行動：** 修訂 FR-MM-05 為：「處於 LIVE 模式超過 24 小時後，於下一次系統啟動或排程觸發時，自動切回 SIMULATION 並寄出告警 Email。」

### I-7 新聞抓取時區與 DST（夏令時間）── **缺漏**
FR-NS-01：「每日於台北時間 06:00（美股盤後）與 21:30（美股盤前）」。
- 美股有夏令時間（DST），盤後時間在 04:00 ~ 05:00 台北時間之間切換 ± 1 小時。
- 06:00 台北抓取是否一定能拿到完整盤後新聞？21:30 抓取是否能拿到完整盤前新聞？

**建議行動：** 補 FR-NS-35：抓取時間需依 DST 自動調整（例如改為「美股盤後 1.5 小時後抓取」這種相對時間定義）。

### I-8 資料庫備份與還原 ── **完全缺漏**
- 本系統高度仰賴 SQLite（持倉、訂單、訊號、新聞、LLM 分析、PnL 全部在裡面）。
- 沒有任何備份、還原、匯出機制的需求。
- 若 `%LOCALAPPDATA%\StocksTrading\app.db` 損毀（SSD 壞軌、誤刪、Windows 升級）── 全部歷史資料消失。

**建議行動：** 補 NFR-REL-04 / NFR-DEP-05：
- 每日策略執行後自動備份 SQLite 至 `%LOCALAPPDATA%\StocksTrading\backups\YYYYMMDD.db.gz`。
- 設定頁提供「立即備份」、「從備份還原」按鈕。
- 保留最近 30 天 + 月底快照 12 個月。

### I-9 MSI 升級保留資料 ── **規格不夠細**
NFR-DEP-04 只說「升級保留使用者資料」。實務上需要明確：
- 哪些檔案保留：`config.json` / `secrets.dat` / `app.db` / `backups/` / `logs/`？
- schema migration 規則？（v0.2 → v0.3 加了新欄位，舊資料如何 migrate？）
- 若使用者手動移除舊版再裝新版（非升級），資料夾如何處理？

**建議行動：** 補 NFR-DEP-06：MSI 升級時 `%LOCALAPPDATA%\StocksTrading\*` 完全不動；升級後第一次啟動執行 schema migration（使用 sqlalchemy alembic 或自家版本表）。

### I-10 Watchlist「轉為交易訊號」的下單時機 ── **未定義**
FR-NS-22：點擊「轉為訊號」後通過 RiskGuard 並下單。但：
- 是「立即」下單嗎？還是「下次排程觸發」時下單？
- 美股若處於盤後時段才核可，是寄信給使用者立即手動下，還是等下次盤前？
- 美股實盤 ── 是另外寄一封新訊號 Email，還是進入下次每日摘要？

**建議行動：** 補 FR-NS-36：Watchlist 轉訊號後的時機為「立即」（呼叫對應 Broker），美股 LIVE 立刻寄出獨立訊號 Email，但 Email 主旨需與每日策略訊號區分（例如 `[SIM] 新聞訊號 - 2026-05-23 - 買進 NVDA`）以便篩選。

### I-11 模擬模式的「成交價」── **規則模糊**
FR-EX-03 寫「以日線收盤價（或隔日開盤價）成交」。
- 究竟是收盤價還是開盤價？「或」字製造了不確定性。
- 用收盤價成交 = 偷看未來（look-ahead bias），回測結果會虛胖。
- 用隔日開盤價成交 = 合理（無 look-ahead），但需要等 T+1 才有成交。

**建議行動：** 修訂 FR-EX-03 為：模擬模式以「訊號產生日的隔日開盤價」成交（避免 look-ahead bias），扣除手續費與滑價（預設手續費 0.1425% 台股 / 0.18% 美股、滑價 0.05%）。

### I-12 LLM 預算重置時點 ── **規則模糊**
FR-NS-10 / 11：每日 LLM 成本上限 $0.3 USD。
- 「每日」是依台北時間？UTC？還是 Claude API billing 的時區？
- 跨日重置是 00:00 還是 06:00（與新聞抓取對齊）？

**建議行動：** 補 FR-NS-37：LLM 預算重置時點為台北時間 00:00；`llm_cost_daily.date` 欄位以台北時區計算。

### I-13 「測試信」與生產 Email 共用收件人風險 ── **小議題**
FR-NT-04 提供「寄送測試信」按鈕。HTML mockup 設定頁的收件人就是 gamma.jeff.lin@gmail.com（與寄件人同一個信箱）。
- 沒有規定測試信主旨應加 `[TEST]` 標記。
- 若使用者測試完忘記，後續真實訊號信也會夾雜在同一收件人收件匣中，難以區分。

**建議行動：** 補 FR-NT-09：測試信主旨需以 `[TEST]` 前綴，與正式 `[SIM]` / `[LIVE]` / `[ALERT]` 區別。

### I-14 整份文件未提「日誌保留與滾動」── **缺漏**
- `app_log` 表會持續增長。
- 沒有規定保留期限、輪替策略、檔案大小上限。
- 一年下來可能數十萬筆。

**建議行動：** 補 NFR-MNT-04：`app_log` 預設保留 90 天，每日自動清理；可於設定頁調整。同時提供「匯出近 N 天日誌為 ZIP」按鈕便於 debug。

## 2. 建議補強（enhancement）

下面這些不是「現有需求有問題」，而是「應該加但目前完全沒寫」的需求。

### E-1 應用程式更新檢查與通知
- MVP 是個人用 MSI，沒有自動更新機制。
- 但若日後修 bug、改演算法 ── 使用者怎麼知道有新版？
**建議：** 補 FR-UI-14 / NFR-DEP-07：應用程式啟動時檢查 GitHub Release（或自定義 endpoint）是否有新版，於設定頁顯示「目前 v0.2 · 最新 v0.3 可下載」。

### E-2 第一次啟動 onboarding wizard
- UC-01 描述安裝後設定頁輸入帳密 ── 但沒有引導式設定流程。
- 對「同時是開發者」的使用者來說可能不重要，但若日後分享給朋友會很重要。
**建議：** P2 等級，補 FR-UI-15：首次啟動顯示 5 步驟設定精靈（Shioaji → SMTP → 風控 → 策略 → 完成）。

### E-3 操作審計日誌（audit log）
- `app_log` 是系統技術日誌（連線、錯誤等）。
- 但「使用者於 14:23 手動觸發 SIM → LIVE 切換」這類操作軌跡同樣重要。
**建議：** 補 NFR-SEC-05：所有「敏感操作」（模式切換、風控參數調整、設定變更、Watchlist 轉訊號）需寫入獨立的 `audit_log` 表，保留 365 天不可刪除。

### E-4 模擬模式績效歸因
- 回測結果有 Sharpe / 最大回撤 / 勝率 ── 但模擬模式累積跑 3 個月（M7）後，沒有同等的績效歸因報表。
- M8 上實盤前應該先比對「模擬 3 個月 vs 同期回測 3 個月」結果是否一致。
**建議：** 補 FR-BT-06：提供「實盤 / 模擬 / 回測」三組同期間績效對比報表，幫助驗證實盤偏差。

### E-5 LLM 成本歷史趨勢
- 目前只記錄當日累計（`llm_cost_daily`）。
- 沒有 UI 顯示「過去 30 天 LLM 花費趨勢」、「平均每篇成本」、「成本最高的來源」。
**建議：** 補 FR-NS-38：新聞情緒分頁提供「成本分析」子分頁，顯示 30 天成本趨勢與每篇平均成本。

### E-6 Watchlist 過期前提醒
- FR-NS-25：Watchlist 7 天後自動過期。
- 若使用者忘記核可，重要候選會悄悄消失。
**建議：** 補 FR-NS-39：Watchlist 項目過期前 24h 在 GUI 顯示「即將過期」標記；可選擇延長至 14 天。

### E-7 黑名單覆蓋優先級
- FR-NS-23：黑名單可永久排除 ticker 或 source。
- 但若 NVDA 在黑名單，新聞模組抓到 NVDA 文章 ── 是 (a) 不分析？(b) 分析但不進候選？(c) 進候選但 UI 標示黑名單？
**建議：** 補 FR-NS-40 明定黑名單規則：黑名單 ticker → 候選階段過濾（分析仍進行，留作歷史檢視）；黑名單 source → 抓取階段過濾（連分析都不做，節省 LLM 成本）。

### E-8 多語言新聞策略明確化
- 第 13 章開放議題 #8 提到中文新聞。
- MVP 暫不做沒問題，但需要：(a) 抓取時過濾非英文文章？(b) 抓到中文文章如何處理？
**建議：** MVP 補 FR-NS-41：Collector 階段以語言偵測過濾，非英文文章直接捨棄（不計入 LLM 成本）；中文新聞計畫於 v0.3 後評估。

## 4. 依賴關係風險（里程碑順序）

目前里程碑：M0 → M1 → M2 → M3 → M4 → M5 → M5.5 → M5.7 → M6 → M7 → M8。

### 4.1 M3 GUI 與 M4 通知的順序
- M3（GUI 2 週）排在 M4（通知 0.5 週）之前。
- 但 M3 的「設定頁」需要包含 SMTP 設定欄位 + 測試信按鈕（FR-NT-04），這要求 M4 通知模組先做出來。
- 建議：M4 拆成兩段：「M4a SMTP 核心 0.3 週」做在 M3 之前；「M4b Email 範本與告警 0.2 週」做在 M3 之後。

### 4.2 M5.5 新聞模組依賴
- 「新聞情緒模組依賴 M3 GUI、M5 實盤整合」── 這個順序大致對。
- 但 M5.5 真的需要 M5 實盤整合嗎？新聞模組本身不下單，只進 Watchlist。
- 結論：M5.5 **只強依賴 M3（要有 GUI 顯示 News 分頁）+ M2（要有 RiskGuard 用於 Promote → Signal 後的審查）**。M5 不是強依賴。
- 建議：M5.5 可以與 M5 並行，或排在 M5 之前。

### 4.3 M5.7 K 線圖表依賴
- M5.7 列在 M5.5 之後 ── 但 K 線圖表的核心需求（FR-CH-01 ~ FR-CH-16）只依賴 M1（資料）+ M3（GUI），與 M5 / M5.5 無關。
- 唯一的耦合點是 FR-CH-06「持倉 / Watchlist 列尾 sparkline」── 這需要 Watchlist（M5.5）。
- 結論：M5.7 大部分功能可以在 M3 之後直接做，只有 sparkline 那部分需要等 M5.5。
- 建議：將 M5.7 拆成「M5.7a 核心 K 線 1 週」（在 M3 後）與「M5.7b Watchlist sparkline 0.5 週」（在 M5.5 後）。

### 4.4 M7 Paper Trading 與 M5.5 / M5.7 的關係
- M7「Paper Trading 3 個月」是與時間綁定的驗證階段，不是開發工作。
- 但 M5.5 / M5.7 完成是在 M6 之前 ── 而 M6 是打包部署。
- 如果新聞模組要納入 paper trading 驗證，需要先把 M5.5 / M5.7 包進 MSI 給自己用，這就需要 M6 完成。
- 建議：明確標註 M7 開始時應該已經包含 M5.5 + M5.7 + M6，這樣 paper trading 才能真正驗證 v0.2 完整功能。

### 4.5 風險：M8 實盤的「就緒條件」未定義
- M8「實盤小額 1 萬以下」── 觸發 M8 的條件是什麼？M7 paper trading 3 個月 + 達到某 Sharpe 門檻？
- 沒寫退場機制：實盤虧損達多少要回 paper？
**建議：** 補 M8 進入準則：
- M7 跑滿 3 個月不中斷
- Sharpe ≥ 0.8 且最大回撤 ≤ 15%
- 模擬績效偏離回測 < 20%
- 連續 30 天無系統錯誤

退場機制：實盤累計虧損超過 5% 時自動回 paper、寄告警。

## 5. 預估工時合理性

### 5.1 M5.5（新聞情緒模組 2 週）── **嚴重低估**
細看 4.10 涵蓋的範圍：
- Collector：7+ 來源 adapter（yfinance, CNBC RSS, Reuters RSS, Ars Technica, TechCrunch, The Verge, Reddit, EDGAR）── 每個 adapter 都需要：抓取、解析、去重、儲存、錯誤處理。**保守估 2-3 天**。
- LLM Analyzer：Claude SDK 整合 + prompt 設計 + JSON schema 驗證 + prompt caching + 重試 + 成本記錄。**保守估 2 天**。
- TickerMapper：yfinance lookup + 內建對照表 + 信心度演算法。**保守估 1-1.5 天**。
- Ranker：4 因子加權公式、強訊號標示。**保守估 0.5-1 天**。
- WatchlistService：CRUD、過期清理、黑名單。**保守估 1 天**。
- GUI 新聞情緒分頁：新聞 feed、候選清單卡片、成本進度條、Watchlist 兩段核可、原文 viewer、查看 K 線跳轉。**保守估 3-4 天**。
- Daily Digest Email 範本：HTML + 強訊號分區 + 弱訊號列表。**保守估 1 天**。
- 整合測試（含模擬 LLM 回應、mock API、邊界情況如成本超過上限）。**保守估 1 天**。

合計：**12-15 個工作天 ≈ 2.5 ~ 3 週**。預估 2 週偏低 50%。

**建議：** M5.5 改為 **3 週**，或將 P2 項目（FR-NS-18 / FR-NS-24）延後到 v0.3。

### 5.2 M5.7（K 線圖表 1.5 週）── **偏低估**
- pyqtgraph K 線元件：自訂蠟燭、互動縮放、十字游標、tooltip ── pyqtgraph 不像 mplfinance 開箱即用，需大量自寫。**2-3 天**。
- 5 種 timeframe 切換 + 資料重採樣。**0.5-1 天**。
- 5 條 MA + 布林 + RSI + MACD + 成交量副圖 + 統一 IndicatorEngine。**1.5-2 天**。
- 5 種 PatternDetector + 標籤標示於 K 線。**1.5 天**。
- 圖表分頁 + 指標 chip toggle + 右側形態列表 + 當前指標表。**1.5 天**。
- 持倉 / Watchlist sparkline 元件、回測進出場視覺化。**1 天**。
- 主題切換相容性測試（深色 K 線、深色 MA 線顏色對比）。**0.5 天**。

合計：**8.5-11.5 個工作天 ≈ 1.7 ~ 2.3 週**。預估 1.5 週偏低約 30%。

**建議：** M5.7 改為 **2 週**；或暫時砍掉 FR-CH-27 （形態作為過濾器，本來就標 P2）。

### 5.3 M3（GUI 2 週）── **可能低估**
PySide6 五大分頁 + 模式切換 + 設定頁，看似 2 週合理，但：
- 設定頁要做 DPAPI 加密整合、欄位驗證、測試按鈕、儲存重啟邏輯。
- 主題切換需要 ≥ 70% 元件支援深色版（FR-UI-08「即時切換不需重啟」是高難度）。
- KPI 卡片、持倉表、訊號表、績效曲線、子分頁、系統狀態面板。
- 加入第 2 節 implicit requirements 後，工作量會再增 20-30%。

**建議：** M3 改為 **2.5 ~ 3 週**。

### 5.4 M6（打包 0.5 週）── **嚴重低估**
- cx_Freeze 打包 PySide6 + Shioaji（含 native dll） + ta-lib（含 C 編譯）── 這些都是 cx_Freeze 的踩坑大戶。
- MSI 安裝程式：捷徑 / 開始選單 / 卸載 / 升級保留資料。
- 簽署（自簽 + Windows SmartScreen 提示）。
- 第一次安裝測試 + 升級測試 + 解除安裝測試。
- Task Scheduler 設定範本（.xml 或腳本）。

**建議：** M6 改為 **1 ~ 1.5 週**。打包是經典「以為很快結果踩雷」的階段。

## 6. 高優先級行動清單（給開發者本人）

## 7. 結論

文件已具備進入 **設計階段** 的基礎，但建議先針對 I-1 / I-2 / I-3 / I-10 / I-11 五個關鍵業務規則做一次「需求釐清會」（即便是自己跟自己），形成 v0.3，再進入 M0 開發。

工時上建議實際排程為 **15-16 週（含 buffer）** 而非原訂 10.5 週；M7 / M8 屬於驗證階段，不算開發時間。

最後一個提醒：**這份文件對「使用者過度依賴 LLM」這條風險已經有警覺（11.1 表）── 但同樣的提醒也適用於開發者本人 ──「跑 3 個月模擬績效漂亮就上實盤」是很多自動化交易系統的死因。建議在 M8 進入條件中明確列入「至少經歷一次回撤事件且 RiskGuard 確實攔截過訊號」**，以驗證風控機制真的有效，而非靠運氣度過順風期。
